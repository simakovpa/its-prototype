# Прототип модуля ИТС — «Шаблон расчёта»

Демонстрационное React-приложение (Vite + Ant Design 5), реализующее рекурсивный
движок расчёта индекса технического состояния по решениям из
`its_module_architecture_decisions.md`.

## Запуск

```bash
npm install
npm run dev
```

Откроется на `http://localhost:5173`.

## Публикация на GitHub Pages

```bash
npm run deploy   # соберёт dist/ и скопирует в docs/
git add docs -f
git commit -m "Update prototype"
git push
```

В Settings → Pages репозитория: источник — Deploy from branch, ветка `master`, папка `/docs`.

## Библиотека типовых узлов и шкал

Отдельная вкладка «Библиотека» — самостоятельный экран, не привязанный ни к одной
конкретной методике. Решает пробел, который не закрывали ни «Вставить из другой
методики» (нужно помнить, в какой методике что-то похожее уже настроено), ни
«Добавить группу оборудования» (это про целые методики, а не про фрагменты):

- **Типовые узлы** — два способа наполнения: (1) кнопка «Создать» прямо на экране
  «Библиотека» — с нуля, дальше редактируется тем же деревом, что и обычная
  методика; (2) в редакторе методики выбрать уже настроенный узел и «Сохранить
  в библиотеку». Вставка в методику — кнопкой «Вставить из библиотеки», всегда
  независимая копия.
- **Типовые шкалы** — аналогично: «Создать» на экране библиотеки или «Сохранить
  шкалу в библиотеку» из формы параметра. Загружаются в параметр через
  «Загрузить из библиотеки» в форме параметра. Узлы и шкалы библиотеки —
  независимые полки, между собой не связаны: шкала внутри сохранённого узла —
  встроенная копия на момент сохранения, а не ссылка на запись библиотеки шкал.

Тем самым в системе теперь три разных по природе способа переиспользования —
и это осознанный выбор, а не три случайно расплодившихся механизма:

| Способ | Что переиспользуется | Связь с оригиналом |
|---|---|---|
| Библиотека (узлы/шкалы) | Курируемый, самостоятельный фрагмент | Независимая копия |
| «Вставить из другой методики» | Что угодно, что уже где-то настроено | Независимая копия |
| «Добавить группу оборудования» | Целая методика оборудования/ТМ | Живая связь — изменения в связанной методике сразу отражаются везде, где она подключена |

## Реестр методик

Методики (шаблоны расчёта) больше не жёстко зашиты — это список (`src/data/methodologyTemplates.js`,
`initialMethodologies`), которым управляет пользователь через кнопку «Добавить методику» на вкладке
«Шаблон расчёта ИТС». У каждой методики есть уровень применения (Оборудование / ТМ / Объект) и код
типа актива, к которому она относится.

При редактировании методики уровня Объект/ТМ доступны два способа переиспользования уже настроенного:

- **«Вставить из другой методики»** — копирует выбранный узел (со всем содержимым) из любой другой
  методики в текущую. Это независимая копия, дальше её можно редактировать без риска задеть оригинал.
- **«Добавить группу оборудования»** — создаёт не копию, а **связанную** динамическую группу: на
  расчёте система сама находит все единицы выбранного типа на объекте и считает их ИТС по связанной
  методике оборудования. Если методику оборудования потом изменить — это сразу отразится на расчёте
  объекта, без необходимости что-либо трогать в методике объекта.

## Что реализовано

- **Рекурсивный движок** (`src/engine/calculate.js`, `strategies.js`) — одна и та же
  сущность «Узел методики» (`kind: 'container' | 'leaf'`), работает на любом уровне:
  Этап → Узел → Группа → Параметр, а также на уровне Объекта (динамические группы
  оборудования + технологическая цепочка).
- **Каталог стратегий свёртки**: MIN, простое среднее, взвешенная сумма,
  взвешенный дефицит-индекс (баллы 0–4 → индекс 0–100), взвешивание по показателю
  приведения, условная свёртка по доле «плохих» элементов.
- **Три источника данных параметра**: измеряемый параметр (с абсолютным значением,
  отклонением от Фо, трендом по N значениям), дефект (явный список типов из
  справочника, без зависимости от поля тяжести), паспортное значение (возраст
  относительно нормативного срока службы).
- **Материализация узлов**: виртуальные (параметры на самой ЕО) и материализованные
  (ссылка на ЕО-сиблинга, например «Ввод» — со сменой контекста расчёта на
  дочернюю ЕО).
- **Обработка отсутствия данных**: конфигурируемое поведение параметра (балл 0 /
  исключение с перенормировкой весов) и отдельно — структурное отсутствие
  (материализованный узел не найден) с автоматической перенормировкой.
- **Правила коррекции** — на любом нелистовом уровне, пример: капитальный порог
  индекса при критическом состоянии электрической части (п. 3.8 Методики).
- **Категории состояния** — по таблице №1 Приказа №676 (критическое ≤25 …
  очень хорошее до 100), с отдельной категорией «Не определено».
- **Редактор шаблона** (вкладка «Шаблон расчёта ИТС») — дерево с добавлением/
  удалением Этапов/Узлов/Групп/Параметров и формой свойств для каждого узла.
- **Тестовый и официальный расчёт** (вкладка «Тестовый / официальный расчёт») —
  пробный прогон не пишется в историю, официальный — попадает в таблицу истории
  (аналог сущности «Значение ИТС»); отдельно — расчёт по объекту (технологическая
  цепочка, агрегирующая уже посчитанные ЕО по показателю приведения — мощности).

## Моковые данные (`src/data/`)

Одна подстанция, два трансформатора (Т-1 — старый, с двумя открытыми дефектами;
Т-2 — новый, исправный), у каждого — свой Ввод, несколько протоколов измерений
(в т.ч. для проверки тренда к Фо) и справочники (номенклатура, типы дефектов,
измеряемые параметры).

## Сознательно упрощено / не реализовано в этой версии

Смотрите открытые вопросы (раздел 11) архитектурного документа — почти всё, что
там помечено «отложено», в прототипе тоже не реализовано: роли и права доступа,
модуль «Аналитика», UI-полировка форм (drag-and-drop сортировка детей, валидация
суммы весов), полноценные технологические цепочки для ТП/ВЛ, состав статусов
дефекта. Само устройство расчёта (то, ради чего строился прототип) реализовано
полностью в соответствии с документом.
